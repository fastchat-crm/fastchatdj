{# Modal Tutorial — guía nivelada para crear y operar un flujo #}
Tutorial — Cómo crear y operar un flujo de chatbot
¿Qué es un flujo? Un departamento contiene una secuencia de nodos que el chatbot ejecuta cuando un cliente entra. Cada nodo pregunta algo, muestra un menú, llama a una API o termina la conversación. El flujo se enlaza con conexiones (qué nodo viene después según la respuesta del cliente, el éxito de un HTTP, etc.).
{# ── NIVEL BÁSICO ─────────────────────────────────── #}
Nivel 1 — Básico (lo mínimo para arrancar)
  • Cabecera del depto: nombre, color, mensaje de saludo. El primer guardado crea el depto.
  • Crear un nodo: botón Agregar opción raíz → modal con tipo + nombre + config.
  • Marcar el inicio: poné es_inicio=true en el nodo desde donde arranca el flujo.
  • Encadenar nodos: en el modal de nodo, sección "Conexiones" → elegí el siguiente nodo (etiqueta vacía = default).
  • Terminar: usá un nodo tipo fin con un mensaje de despedida. Aparece marcado con Termina flujo.
  • Probar: botón Preview simula el chat sin tocar WhatsApp.
{# ── NIVEL INTERMEDIO ──────────────────────────────── #}
Nivel 2 — Intermedio (variables, validaciones, menús)
  • Capturar respuestas: nodo tipo pregunta con variable_destino (ej. cedula) → el contenido del cliente queda en {% verbatim %}{{variables.cedula}}{% endverbatim %}.
  • Validaciones: regex (^\d{10}$), email, teléfono — el bot reintenta hasta reintentos_max y luego salta a la rama de error.
  • Menús: nodo tipo menu → opciones estáticas (etiqueta → siguiente) o dinámicas (opciones_fuente = lista cargada desde una variable).
  • Condicionales: nodo tipo condicional con expresiones tipo {% verbatim %}{{variables.encontrado}} == true{% endverbatim %} → ramas true/false.
  • Asignar variables: set_variable permite calcular o resetear valores. El último paso suele resetear todo antes del fin.
  • Plantillas: en cualquier mensaje podés interpolar variables: {% verbatim %}{{contacto.nombre}}{% endverbatim %}, {% verbatim %}{{conversacion.id}}{% endverbatim %}, {% verbatim %}{{variables.foo}}{% endverbatim %}.
{# ── NIVEL AVANZADO ────────────────────────────────── #}
Nivel 3 — Avanzado (HTTP, side-effects, integración)
  • Llamadas HTTP: tipo http con endpoint registrado en /crm/endpoints_api/. Path soporta templates: api/cotizar/{% verbatim %}{{conversacion.id}}{% endverbatim %}/.
  • Extraer variables: extraer = [{variable: 'cotpk', jsonpath: 'data.cotpk'}] — JSONPath relativo al body de respuesta.
  • Ramificación ok/error: status 2xx con success: true → rama ok; 4xx/5xx o success: false → rama error.
  • Envío de correo: marcá el nodo con config.envia_correo = true y el sistema lo identificará con Envía correo. Útil para procesos como "notificar asesor cuando se cotiza".
  • Proxy interno: para side-effects que requieren lógica Django (DB lookup + email + log), apuntá a un endpoint propio (/crm/api/cotizar/<conv>/) que orquesta todo.
  • Handoff humano: tipo handoff pausa la IA y deja la conv abierta para que un asesor responda manualmente.
  • Deep-links: dentro de un correo o CTA podés generar /whatsapp/conversaciones/?conv=<token> con encrypt_sesion_id(conv.id) — abre la conv si está activa, o la página de finalizadas si ya cerró.

{# ── BUENAS PRÁCTICAS ──────────────────────────────── #}
Buenas prácticas
  • Numerá los nodos en saltos de 10 (10, 20, 30…) para insertar pasos intermedios sin renumerar.
  • Toda rama de error debería desembocar en un fin con un mensaje claro tipo "Inténtalo más tarde".
  • Antes del fin, usá set_variable para limpiar las variables temporales (cedula, placa, etc.) por si el cliente reabre la conv.
  • En nodos HTTP, configurá timeout_seg realista — un POST a un cotizador puede tardar 30-60s.
  • Las URLs externas y los mensajes con {% verbatim %}{{variables.foo}}{% endverbatim %} se resuelven en runtime — probá con Preview antes de exponerlo.
  • Si un nodo dispara correos / notificaciones, marcá config.envia_correo = true para que el operador lo vea como tal en el editor.
