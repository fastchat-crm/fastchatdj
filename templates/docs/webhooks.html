Webhooks

Un webhook es un aviso automático entre sistemas vía HTTP. fastchat usa dos tipos: entrantes (otros nos avisan, ej. llegó un mensaje de WhatsApp) y salientes (nosotros avisamos a apps externas cuando pasa algo).

Webhooks entrantes

Otros sistemas nos mandan un POST. Así recibimos mensajes y eventos.

# URL Quién lo llama Función
1 /whatsapp/webhook_handler/ Node Baileys Eventos de WhatsApp no oficial: mensajes entrantes, código QR, conexión lista, confirmaciones (ACK).
2 /whatsapp/webhook_handler/batch/ Node Baileys Igual que el anterior pero recibe varios eventos en lote (más eficiente).
3 /whatsapp/meta_webhook/ Meta Cloud API WhatsApp oficial: mensajes, estados (enviado / entregado / leído) y cambios de estado de plantillas. Valida firma HMAC e identifica la sesión por phone_number_id.
4 /whatsapp/instagram_webhook/ Meta (Instagram) Mensajes directos (DM) de Instagram.
5 /whatsapp/messenger_webhook/ Meta (Messenger) Mensajes de Facebook Messenger.
6 /whatsapp/heartbeat/ Node Baileys "Latido": el servicio Node avisa periódicamente que sigue vivo, para detectar caídas de sesión.
7 /whatsapp/trace/ WebService externo Recibe trazas / logs personalizados de un proceso externo, para auditoría.
Los webhooks de Meta llegan firmados con X-Hub-Signature-256 (HMAC-SHA256). fastchat valida la firma con el app secret antes de procesar; si no coincide, rechaza el evento.

Webhook saliente

fastchat manda un POST a una URL externa cuando ocurre un evento. Es el "Zapier nativo".

# Qué Función
8 Webhook saliente configurable La URL la define el cliente (no es fija). Se suscribe a eventos (ej. conversacion.nueva, mensaje.entrante) y recibe el JSON cuando ocurren.

¿Para qué sirve?

  • Crear automáticamente el lead en tu CRM / sistema de matrículas.
  • Enviar los datos a Google Sheets, Zapier, Make o n8n sin programar.
  • Avisar a ventas por Slack / Discord cuando entra un interesado.
  • Disparar facturación cuando una conversación se marca como cliente.

Confiabilidad (backoff)

Si la URL externa está caída, fastchat reintenta con espera creciente (1 min → 2 → 4… hasta un tope de 6 h) y, tras 8 fallos seguidos, desactiva el webhook para no afectar al sistema. Un envío exitoso reinicia el contador. Cada intento queda registrado. Si definís un secret, el cuerpo va firmado con HMAC en el header X-FC-Signature.

El motor del webhook saliente y su backoff ya están construidos. Para activarlo de punta a punta falta enlazarlo a los eventos del sistema (un paso de configuración técnica).

Auditoría (no son webhooks)

Estas son pantallas del panel para ver los webhooks, no receptores.

  • /whatsapp/sesiones/<id>/webhook-log/ — eventos Meta recibidos por sesión (procesados / con error).
  • /whatsapp/meta/webhook-hits/ — hits HTTP crudos del webhook Meta (status, IP, firma, cuerpo) para depuración técnica.

Resumen

  • Recibir mensajes de clientes → webhooks 1 a 5 (según canal / proveedor).
  • Salud del Baileys → webhook 6 (heartbeat).
  • Logs externos → webhook 7 (trace).
  • Integrar con otras apps → webhook 8 (saliente).
