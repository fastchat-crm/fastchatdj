FastChat DJ

Plataforma CRM de WhatsApp con agentes de inteligencia artificial, gestión de conversaciones en tiempo real y automatización de atención al cliente.

Django 4.x LangChain FAISS Google Gemini OpenAI Django Channels Redis PostgreSQL Node.js

¿Qué es esta plataforma?

FastChat DJ es un CRM web construido sobre Django que permite gestionar múltiples sesiones de WhatsApp, centralizar conversaciones entrantes, y automatizar respuestas mediante agentes de IA entrenados con documentos propios del negocio.

Está diseñado para empresas que atienden clientes por WhatsApp y desean escalar sin aumentar el equipo humano, usando LLMs (Gemini o GPT) combinados con una base de conocimiento personalizada.

Módulo: Sesiones WhatsApp

Cada SesionWhatsApp representa un número de teléfono conectado. La plataforma se comunica con un servicio Node.js externo que maneja la conexión real con la API de WhatsApp.

Ciclo de vida de una sesión

1
Crear sesión en el panelSe registra en la BD con estado "desconectado"
2
Conectar → QR via WebSocketEl Node.js genera el QR, lo envía al panel via SessionConsumer
3
Escanear QR con el celularWhatsApp autentica → evento ready actualiza el estado a "conectado"
4
Mensajes entrantes via webhookPOST /whatsapp/webhook_handler/ procesa todos los eventos

Configuración del agente IA en la sesión

Cada sesión puede tener asignado un AgentesIA. Si está configurado, cada mensaje entrante dispara automáticamente una consulta al agente y envía la respuesta al cliente.

Si la sesión no tiene agente asignado, los mensajes solo se almacenan y muestran en el panel para respuesta manual.

Módulo: Contactos y Conversaciones

Los Contacto se crean automáticamente al recibir el primer mensaje de un número. El sistema sincroniza el nombre de perfil de WhatsApp.

ConversacionWhatsApp

Cada par (sesión, contacto) tiene una conversación única. Los MensajeWhatsApp se asocian a la conversación. El ChatConsumer (WebSocket) envía actualizaciones en tiempo real al panel cuando llega un nuevo mensaje.

Campo clave Descripción
estado abierta / en_proceso / cerrada — controlado por reglas de cierre
leido Flag de lectura para el agente humano
ai_activo Si el agente IA responde en esta conversación (puede desactivarse por turno)
EstadisticasConversacion Tiempo de respuesta, total mensajes, primer contacto, etc.

Módulo: Agentes IA

Un AgentesIA es la configuración completa de un chatbot. Se puede reutilizar en varias sesiones.

Campo Función
nombre Identificador del agente
perfil Vincula a PerfilNegocioIA: descripción del negocio, productos, servicios
vectorstore_path Ruta al índice FAISS en disco (construido desde documentos de entrenamiento)
contexto_estatico Texto inyectado directamente al prompt sin pasar por FAISS (ideal para FAQs cortas)
prompt_template Plantilla Jinja con variables {context}, {question}, {historial}, {descripcion_agente}
apikey Relación M2M a ApiKeyIA (Gemini o OpenAI)
departamento Agrupación lógica para organizaciones con múltiples áreas

Lógica de respuesta del agente

  1. El mensaje del usuario llega al webhook
  2. Se carga AgenteConsultor con la config del agente
  3. Si el mensaje es un ack corto ("ok", "gracias", "sí") se omite la búsqueda vectorial
  4. Se construye la query de búsqueda combinando el mensaje + las últimas 3 frases del historial + snippet de la última respuesta IA
  5. FAISS devuelve los 5 chunks más relevantes (MMR, pool de 20) de máximo 2.500 caracteres
  6. Se ensambla el prompt con contexto, historial y pregunta
  7. El LLM genera la respuesta
  8. Se registran los tokens en ConsumoTokenIA
  9. La respuesta se envía al cliente por WhatsApp
Sin datos de entrenamiento: Si el agente no tiene vectorstore ni contexto estático, responderá indicando que no tiene información. Es esencial cargar al menos un documento o agregar contexto estático antes de activar el agente en producción.

Módulo: Entrenamiento

Los documentos de entrenamiento son archivos que se convierten en embeddings vectoriales y se almacenan en un índice FAISS local.

Formatos soportados

PDF Excel (.xlsx) CSV JSON

Proceso de indexación

Archivo subido
    ↓ VectorStoreManager.build_from_files()
    ↓ Extracción de texto por tipo de archivo
    ↓ Chunking (segmentación en fragmentos)
    ↓ Embeddings (Google Gemini o OpenAI)
    ↓ FAISS index guardado en vectorstore_path
    ↓ Cache en memoria invalidado (mtime check)
El índice FAISS se cachea en memoria con verificación de mtime para no reconstruirlo en cada mensaje. El cache se invalida automáticamente al reentrenar.

Contexto estático vs. FAISS

Tipo Cuándo usarlo Ventaja
Contexto estático FAQs, instrucciones cortas, información que siempre aplica Cero latencia, siempre visible para el LLM
FAISS vectorial Documentos extensos: catálogos, manuales, políticas Escala a miles de páginas sin saturar el prompt

Módulo: Detección de Cierre

La detección de cierre permite identificar automáticamente cuando un cliente indica que terminó la conversación ("gracias, hasta luego", "listo, eso era todo", etc.) y ejecutar acciones automáticas.

Configuración por niveles

Nivel sesión

Configurado en la sesión específica de WhatsApp. Tiene prioridad absoluta sobre el agente.

Nivel agente (plantilla)

Si la sesión no tiene cierre configurado, se usa la plantilla del agente asignado.

Acciones disponibles al detectar cierre

Tipo de acción Descripción Variables de plantilla
Mensaje automático Envía un mensaje de despedida personalizado al cliente {nombre}, {telefono}, {fecha}
Cambiar estado Cambia el estado de la conversación (ej: cerrada, en espera) —
Asignar usuario Reasigna la conversación a un agente humano específico —
Webhook externo Llama a una URL externa con datos de la conversación (CRM, ERP) —

Módulo: Consumo de Tokens

El modelo ConsumoTokenIA registra cada llamada al LLM con los tokens utilizados, permitiendo monitorear costos por API key y agente.

Datos registrados por llamada

Campo Descripción
apikey API key usada (FK a ApiKeyIA)
agente Agente que generó la respuesta
tokens_entrada Tokens del prompt enviado al LLM
tokens_salida Tokens de la respuesta generada
tokens_total Suma entrada + salida
modelo Nombre del modelo usado (ej: gemini-1.5-flash)
fecha Timestamp automático de la llamada
Desde la pantalla de API Keys del agente, el botón "Consumo" abre un modal con resumen por rango de fechas, totales del período, promedio diario y desglose por agente en barras.

API REST

La plataforma expone un endpoint HTTP para integración con sistemas externos.

Enviar mensaje

POST /api/enviar-mensaje/
Content-Type: application/json

{
    "session_id": "abc123",
    "phone": "593991234567",
    "message": "Hola, tu pedido está listo"
}
Detalle Valor
Autenticación Requiere sesión Django activa (autenticación de usuario)
Rate limit 30 solicitudes/minuto por IP
Validaciones Sesión activa, contacto existente en el sistema
Respuesta exitosa {"status": "ok", "message_id": "..."}

WebSockets

Django Channels maneja tres consumers WebSocket que permiten actualizaciones en tiempo real sin recargar la página.

Consumer URL Función
ChatConsumer ws/chat/<conv_id>/ Transmite mensajes nuevos renderizados en HTML al panel del agente humano
SessionConsumer ws/session/<session_id>/ Estado de la sesión WhatsApp: QR code en tiempo real, conectado/desconectado
SessionRoomConsumer ws/session-room/ Dashboard global de estado de todas las sesiones
Para WebSockets en producción se requiere Daphne como servidor ASGI y Redis como channel layer. python manage.py runserver no soporta WebSockets.

Recomendaciones a futuro

Procesamiento de audio y multimedia

Integrar transcripción de notas de voz (Whisper de OpenAI o Google Speech-to-Text) para que el agente pueda procesar mensajes de audio. Para imágenes, usar modelos multimodales (Gemini Vision, GPT-4V) que describan el contenido antes de responder.

Handoff humano inteligente

Cuando el agente detecta frustración, preguntas repetidas sin respuesta satisfactoria o solicitudes de hablar con un humano, transferir automáticamente a un agente disponible con resumen de la conversación.

Dashboard de analítica

Métricas de conversación: tasa de resolución del bot vs. humano, tiempo promedio de respuesta, preguntas frecuentes no resueltas, satisfacción del cliente (encuesta post-conversación).

Agentes especializados por intención

Implementar un router de intenciones que derive al agente correcto según el tema: soporte técnico, ventas, facturación. Cada agente con su propio vectorstore y prompt.

Seguridad y cumplimiento

Implementar cifrado end-to-end para mensajes almacenados en BD, política de retención de datos configurable, y exportación de datos personales bajo solicitud (GDPR / normativas locales).

Vectorstore en la nube

Migrar FAISS local a Pinecone, Weaviate o pgvector (PostgreSQL) para mayor escalabilidad y facilitar despliegues multi-instancia sin compartir sistema de archivos.

Integración con Meta API oficial de WhatsApp

Migrar del conector actual (Node.js + whatsapp-web.js, basado en reverse-engineering de WhatsApp Web) a la Meta Cloud API (WhatsApp Business API oficial).

  • Sin riesgo de ban por uso no oficial de la plataforma
  • Soporte oficial de Meta con SLA y uptime garantizado
  • Plantillas HSM (mensajes proactivos aprobados por Meta)
  • Confirmaciones de lectura, estados de entrega nativos
  • Botones interactivos, listas, respuestas rápidas
  • Gestión de medios (imágenes, audio, documentos) con URLs permanentes de Meta

Requiere cuenta Meta Business, número verificado y webhook HTTPS público. El webhook de Django ya recibe eventos POST — solo cambia el proveedor externo.
