Mensajería Instantánea

Cómo construir un flujo de chatbot — desde el saludo inicial hasta una integración avanzada con APIs externas + envío de correos. La guía está organizada por niveles: arrancá por el básico y avanzá cuando tengas claros los conceptos previos.

{# Glosario #}

Glosario rápido

ConceptoQué es
DepartamentoUnidad ejecutable del chatbot. Contiene el flujo completo.
NodoPaso individual del flujo. Cada uno tiene un tipo (menú, pregunta, HTTP…).
ConexiónUne un nodo con el siguiente según una etiqueta (ok, error, valor del menú, true/false).
VariableDato capturado durante la conversación. Disponible vía {% verbatim %}{{variables.nombre}}{% endverbatim %}.
EndpointURL externa registrada en Endpoints API que un nodo HTTP puede llamar.
{# Indicadores visuales #}

Indicadores visuales por nodo

En el editor y el diagrama, cada nodo puede mostrar:

  • Termina flujo — el nodo es de tipo fin. Cierra la conversación.
  • Envía correo — el nodo dispara una notificación por email como side-effect (configurado con config.envia_correo: true).

Ejemplo: en el flujo ARIA — Cotizador de seguros, el nodo 340 (POST proxy → webhook ARIA + email asesores) está marcado con ambos indicadores porque dispara una notificación a los asesores del depto.

{# Nivel Básico #}

Nivel 1 — Básico (lo mínimo para arrancar)

  1. Crear el departamento: nombre, color, mensaje de saludo. El primer guardado lo persiste.
  2. Agregar un nodo raíz: botón Agregar opción raíz en el editor.
  3. Marcar el inicio: es_inicio=true en el nodo desde donde arranca el flujo. Solo uno por depto.
  4. Encadenar nodos: en el modal de cada nodo, sección "Conexiones" → elegí el nodo siguiente. Etiqueta vacía es la conexión default.
  5. Terminar: usá un nodo tipo fin con un mensaje de despedida. El editor lo marca con Termina flujo.
  6. Probar: botón Preview simula el chat sin tocar WhatsApp.
{# Nivel Intermedio #}

Nivel 2 — Intermedio (variables, validaciones, menús)

  • Capturar respuestas: nodo tipo pregunta con variable_destino (ej. cedula) → el contenido del cliente queda en {% verbatim %}{{variables.cedula}}{% endverbatim %}.
  • Validaciones: regex (^\d{10}$), email, teléfono — el bot reintenta hasta reintentos_max y luego salta a la rama de error.
  • Menús: nodo tipo menu → opciones estáticas (etiqueta → siguiente) o dinámicas (opciones_fuente = lista cargada desde una variable JSON-array).
  • Condicionales: nodo tipo condicional con expresiones tipo {% verbatim %}{{variables.encontrado}} == true{% endverbatim %} → ramas true/false.
  • Asignar variables: set_variable permite calcular o resetear valores. El último paso suele resetear todo antes del fin.
  • Plantillas en mensajes: cualquier texto puede interpolar:
    • {% verbatim %}{{contacto.nombre}}{% endverbatim %} — nombre del contacto.
    • {% verbatim %}{{contacto.numero}}{% endverbatim %} — número WhatsApp.
    • {% verbatim %}{{conversacion.id}}{% endverbatim %} — id de la conversación actual.
    • {% verbatim %}{{sesion.nombre}}{% endverbatim %} — nombre de la sesión.
    • {% verbatim %}{{variables.foo}}{% endverbatim %} — cualquier variable capturada.
{# Nivel Avanzado #}

Nivel 3 — Avanzado (HTTP, side-effects, integración)

  • Llamadas HTTP: tipo http con un endpoint registrado. El path soporta templates: buscar/{% verbatim %}{{variables.cedula}}{% endverbatim %}/.
  • Funciones internas: tipo funcion ejecuta código Python registrado con @registrar_funcion('codigo') en crm/funciones_chatbot.py. Reemplaza un nodo HTTP cuando la lógica vive dentro de Django (orquestar webhooks externos, DB lookups, side-effects). El operador asocia un EndpointApiChatbot opcional → la URL externa queda 100% configurable sin tocar código.
  • Extraer variables del response: 
    "extraer": [
  {"variable": "cotpk",    "jsonpath": "data.cotpk"},
  {"variable": "cliente",  "jsonpath": "data.cliente_id"}
]
    JSONPath relativo al body de respuesta.
  • Ramificación ok / error:
    • Status 2xx con success: true (o sin ese campo) → rama ok.
    • Status 4xx/5xx, success: false, o timeout → rama error.
  • Envío de correo: marcá el nodo con config.envia_correo: true y el sistema lo identificará con Envía correo.
  • Side-effects con lógica Django: cuando necesités orquestar DB lookup + email + log + webhook externo, usá un nodo funcion apuntando a una función registrada propia (no hace falta crear un proxy HTTP intermedio).
  • Handoff humano: tipo handoff pausa la IA. La conv queda abierta para que un asesor responda manualmente.
  • Deep-links a conversaciones: en un correo o CTA podés generar /whatsapp/conversaciones/?conv=<token> con encrypt_sesion_id(conv.id). La URL abre la conv si está activa, o redirige a la página de finalizadas si ya cerró.

{# Buenas prácticas #}

Buenas prácticas

  • Numerá los nodos en saltos de 10 (10, 20, 30…) para insertar pasos intermedios sin renumerar.
  • Toda rama de error debería desembocar en un fin con un mensaje claro tipo "Inténtalo más tarde".
  • Antes del fin, usá set_variable para limpiar las variables temporales (cedula, placa, etc.) por si el cliente reabre la conv.
  • En nodos HTTP, configurá timeout_seg realista — un POST a un cotizador puede tardar 30-60s.
  • Las URLs externas y los mensajes con {% verbatim %}{{variables.foo}}{% endverbatim %} se resuelven en runtime — probá con Preview antes de exponerlo.
  • Si un nodo dispara correos / notificaciones / webhooks externos, marcá config.envia_correo = true para que el operador lo vea como tal en el editor.
{# Caso completo #}

Caso completo: ARIA Cotizador

El flujo ARIA — Cotizador de seguros (creado con python manage.py seed_cotizador) ilustra los tres niveles:

  1. Básico: saludo → pedir placa → pedir cédula → confirmar.
  2. Intermedio: catálogos (tipos, provincias) en menús dinámicos construidos desde variables JSON traídas con HTTP.
  3. Avanzado: nodo final 340 tipo funcion que ejecuta la función Python registrada cotizar_aria (en crm/funciones_chatbot.py). La función:
    • POSTea al webhook externo (URL leída desde el EndpointApiChatbot asociado al nodo — editable desde /crm/endpoints_api/).
    • El motor del flujo, viendo config.envia_correo=true, notifica a los asesores disponibles del número (configurados en la sesión de WhatsApp vía PerfilSesionWhatsApp) cuando la función retorna ok.
    • La función devuelve {etiqueta: 'ok'|'error', body, status, error} para que el flujo elija rama ok o error.

El nodo 340 está marcado con Envía correo y los nodos terminales (998/999) con Termina flujo.
